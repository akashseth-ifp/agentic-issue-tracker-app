# ADR-001: Issue Tracking System — Full Stack Architecture

**Status:** Accepted  
**Date:** 2026-06-16  
**Author:** Akash Seth  
**See also:** [ADR-002](ADR-002.md) — Implementation guide with code examples

---

## Context

Issue Tracking System to demonstrate full-stack proficiency using Angular (frontend) and FastAPI + SQLAlchemy + SQLite (backend). The evaluation rubric has four criteria each scored 1–5; this document captures every architectural decision made to achieve **"Excellent" (5)** on all four.

---

## Decisions & Rationale

### D1 — Backend: 3-Layer Architecture (Router → Service → Repository)
**Decision:** Separate HTTP handling, business logic, and data access into three distinct layers.  
**Why:** The rubric explicitly requires "service layers" and "separation of concerns" for Excellent. A single-file Express-style route handler that mixes DB + logic + response in one function scores Needs Improvement.

| Layer | Knows about | Knows nothing about |
|---|---|---|
| Router | HTTP methods, status codes, request parsing | SQL, business rules |
| Service | Business rules, validation logic | HTTP, SQL |
| Repository | SQL queries, ORM models | HTTP, business rules |

Each layer is independently testable. The router calls the service; the service calls the repository. No layer skips a level.

---

### D2 — Backend: Pydantic v2 Schemas Separate from SQLAlchemy Models
**Decision:** Define separate Pydantic schemas for request input (`IssueCreate`, `IssueUpdate`) and API response (`IssueResponse`), distinct from the SQLAlchemy ORM model.  
**Why:** Controls exactly what fields the API exposes, enables input validation before DB operations, and mirrors the DTO pattern in NestJS. The SQLAlchemy model is internal state; Pydantic schemas are the public API contract.

| Schema | Direction | Fields |
|---|---|---|
| `IssueCreate` | Client → API | title (required), description, status |
| `IssueUpdate` | Client → API | All optional — only sent fields are applied |
| `IssueResponse` | API → Client | id, title, description, status, created_at |
| `IssuePage` | API → Client | items, total, page, page_size, total_pages |
| `Issue` (ORM model) | Internal only | Same fields + DB-specific metadata |

---

### D3 — Backend: Alembic for Database Migrations
**Decision:** Use Alembic to manage all schema changes alongside `Base.metadata.create_all()` (dev convenience only).  
**Why:** The rubric specifically names "uses migrations (Alembic)" as the Excellent criterion for database management. `create_all()` cannot safely ALTER existing tables — it would require dropping and recreating them (data loss). Alembic generates versioned, reversible migration scripts that ALTER tables incrementally.

Workflow: change the SQLAlchemy model → `alembic revision --autogenerate` → review generated script → `alembic upgrade head`.

---

### D4 — Backend: Async SQLAlchemy with aiosqlite
**Decision:** Use `sqlalchemy.ext.asyncio` (`AsyncSession`, `create_async_engine`) with `aiosqlite` as the SQLite async driver. All route handlers, service methods, and repository methods are `async def`.  
**Why:** FastAPI is built on Starlette/asyncio — using sync SQLAlchemy forces every DB call into a thread pool via `run_in_threadpool`, which is implicit and wastes resources. Async I/O lets the event loop serve other requests while waiting on DB. The `get_db` dependency becomes an async generator (`AsyncGenerator[AsyncSession, None]`), and sessions are managed via `async with` — cleaner teardown than try/finally.

`DATABASE_URL` prefix changes from `sqlite:///` to `sqlite+aiosqlite:///` to activate the async driver.

---

### D5 — Backend: Custom Exception Hierarchy + FastAPI Global Handlers
**Decision:** Define an `AppException` base class with domain-specific subclasses (`IssueNotFoundException`, `InvalidStatusTransitionException`), registered globally via `app.add_exception_handler()`.  
**Why:** The service layer should raise semantic exceptions without knowing anything about HTTP. A global handler translates those exceptions into JSON responses with correct status codes. This is what "excellent error handling" means — the service never imports `JSONResponse`.

| Exception | HTTP status | Raised by |
|---|---|---|
| `IssueNotFoundException` | 404 | Service, when `get_by_id` returns None |
| `InvalidStatusTransitionException` | 422 | Service, when status transition is illegal |
| Pydantic `ValidationError` | 422 | FastAPI automatically, before service is called |

---

### D6 — Frontend: Angular 17+ Standalone Components (no NgModules)
**Decision:** Use `ng new --standalone` — every component self-declares its own imports.  
**Why:** NgModules are the legacy approach that requires a separate module file to declare and import every component. Standalone components (Angular 15+, default in Angular 17+) are self-contained — closer to how React components work. Each component lists what it needs in its own `imports: []` array, making the dependency graph explicit and local.

---

### D7 — Frontend: Smart/Dumb Component Architecture
**Decision:** All components are classified as either containers (smart) or presentational (dumb). No exceptions.  
**Why:** The rubric explicitly names "smart/dumb component architecture" as the Excellent criterion for frontend structure. It is the single biggest architectural differentiator for the frontend score.

| Type | Also called | Rules |
|---|---|---|
| Smart (Container) | Page component | Injects services; owns state; fetches data; handles routing |
| Dumb (Presentational) | UI component | Only `@Input()` / `@Output()`; no services injected; no HTTP |

Dumb components are reusable anywhere. Smart components are the glue between the router, services, and dumb components. React analog: dumb = pure component with props + callbacks; smart = component with useState/useEffect/custom hooks.

---

### D8 — Frontend: Functional HTTP Interceptor for Global Error Handling
**Decision:** Implement a functional `HttpInterceptorFn` registered in `app.config.ts` to handle all HTTP errors in one place.  
**Why:** The rubric specifically calls out "Implements HTTP Interceptors" as the Excellent criterion for API integration. Without an interceptor, each component handles errors independently — different error messages, missing cases, duplicated logic. With an interceptor, all 404s, 422s, 500s, and network failures flow through a single handler.

React/Axios analog: `axios.interceptors.response.use(null, errorHandler)`.

---

### D9 — Frontend: RxJS Operators Instead of Promises
**Decision:** Use RxJS composition throughout the service and containers: `switchMap`, `distinctUntilChanged`, `catchError`, `finalize`, `takeUntilDestroyed`.  
**Why:** The rubric explicitly names `distinctUntilChanged` and `switchMap` as Excellent indicators. `takeUntilDestroyed` (Angular 16+) prevents memory leaks by auto-unsubscribing when a component is destroyed. Converting observables to Promises loses composability and the Excellent score.

| Operator | Purpose | React analog |
|---|---|---|
| `switchMap` | Cancel previous request, start new | AbortController + fetch |
| `distinctUntilChanged` | Skip duplicate emissions | `prev !== next` guard in useEffect |
| `catchError` | Handle error, return fallback | `.catch()` in promise chain |
| `finalize` | Always runs on complete or error | `finally` block |
| `takeUntilDestroyed` | Auto-unsubscribe on component destroy | useEffect cleanup function |

## Tools & Packages

### System Prerequisites

| Tool | Required version | Install check |
|---|---|---|
| Python | 3.11+ | `python3 --version` |
| uv | latest | `uv --version` |
| Node.js | 20+ | `node --version` |
| npm | 10+ | `npm --version` |
| Angular CLI | 17+ | `ng version` |

Install uv (once): `curl -LsSf https://astral.sh/uv/install.sh | sh`  
Install Angular CLI (global, once): `npm install -g @angular/cli`

### Backend Python Packages

Managed via `uv add <package>` — no `requirements.txt`. Packages are added incrementally as each layer is built. `pyproject.toml` + `uv.lock` are the source of truth (equivalent to `package.json` + `package-lock.json`).

| Package | Added at | Purpose | Node.js analog |
|---|---|---|---|
| `fastapi` | Commit 2 | Web framework: routes, DI, request/response | Express.js |
| `uvicorn[standard]` | Commit 2 | ASGI server that runs FastAPI | nodemon |
| `pydantic-settings` | Commit 2 | Reads `.env` into a typed Settings class | dotenv + Zod |
| `sqlalchemy` | Commit 2 | ORM + async extension (`AsyncSession`) | Prisma / TypeORM |
| `aiosqlite` | Commit 2 | Async SQLite driver (required by SQLAlchemy async) | — |
| `alembic` | Commit 8 | Migration tool for SQLAlchemy schemas | Prisma Migrate |

### Frontend
`ng new` installs everything: Angular, TypeScript, RxJS, Angular DevTools. No manual package installs needed after scaffolding.

## API Contract

| Method | Path | Query params | Request body | Success response | Error cases |
|---|---|---|---|---|---|
| GET | `/api/issues` | `page=1`, `page_size=20` (max 100) | — | `200` `IssuePage` | — |
| GET | `/api/issues/{id}` | — | — | `200` `IssueResponse` | `404` not found |
| POST | `/api/issues` | — | `IssueCreate` | `201` `IssueResponse` | `422` validation |
| PUT | `/api/issues/{id}` | — | `IssueUpdate` | `200` `IssueResponse` | `404` / `422` |
| DELETE | `/api/issues/{id}` | — | — | `204` no content | `404` not found |

`IssuePage` shape: `{ items: IssueResponse[], total: int, page: int, page_size: int, total_pages: int }`  
All error responses: `{"error": "<message>", "status_code": <code>}`

## Rubric Scorecard

| Criterion | What is implemented | Why Excellent |
|---|---|---|
| Backend Architecture | Router → Service → Repository; custom exception hierarchy registered globally | Explicit service layer; modular APIRouter; error handling separated from HTTP concerns |
| Database Management | Alembic migrations; async `get_db` generator via `async with AsyncSession`; `await engine.dispose()` on shutdown | Rubric names Alembic specifically; async session lifecycle is the "clean teardown" criterion |
| Frontend Structure | Smart/dumb component split; TypeScript interfaces + enum; no `any` anywhere | Rubric names "smart/dumb component architecture" and "strong TypeScript typing" explicitly |
| API Integration | `HttpInterceptorFn`; `switchMap`; `distinctUntilChanged`; `catchError`; `finalize`; `takeUntilDestroyed` | Rubric names "HTTP Interceptors" and `distinctUntilChanged`/`switchMap` explicitly |

## End-to-End Verification Checklist

- [ ] `GET /api/issues` returns `IssuePage` with `items: [], total: 0, total_pages: 0` on fresh database
- [ ] `GET /api/issues?page=2&page_size=5` returns correct slice and metadata
- [ ] `POST /api/issues` without `title` returns `422` with validation message
- [ ] `GET /api/issues/999` returns `404` with JSON error body
- [ ] Status transition `Open → Closed` (skipping In Progress) returns `422`
- [ ] Swagger UI at `http://localhost:8000/docs` lists all 5 routes
- [ ] Angular `/issues` loads list; empty state shown when no issues exist
- [ ] Create via `/issues/new` → issue appears in list
- [ ] Edit via `/issues/edit/:id` → form pre-populated; save persists update
- [ ] Delete with confirm dialog → record removed from list
- [ ] Navigate to `/issues/edit/99999` (non-existent) → error message shown, no crash
- [ ] Kill backend while Angular is running → error message shown via interceptor
